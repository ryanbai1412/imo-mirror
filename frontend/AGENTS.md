## Key Patterns

### Data Flow
- All data comes from static JSON in `public/data/`.
- Pages call loaders from `utils/data.ts` (e.g. `loadHallOfFame(origin)`).
- Loaders use Cloudflare Workers `ASSETS.fetch` when available, falling back to global `fetch`.
- The `origin` argument comes from `Astro.url.origin` in page frontmatter.

### Routing
- Page filenames like `hall.aspx.astro` produce routes like `/hall.aspx` to match the original imo-official.org URL structure.
- Query params (e.g. `?code=USA`, `?year=2024`) are read via `Astro.url.searchParams`.

### Layout
- Every page wraps content in `<Layout title="..." description="..." path="/...">`.
- Layout handles `<head>`, SEO meta, Open Graph, JSON-LD, sticky header, nav, footer, and the CommandPalette (⌘K search).

### Components
- Small, typed Astro components with `interface Props`.
- No client-side framework (React/Vue/etc.) — all rendering is server-side Astro.
- Interactive behavior (charts, command palette) uses inline `<script>` tags with vanilla JS and Canvas API.

### Individual Contestant Codes (C01, C02, …)
- Some contestants compete as individuals rather than under a country flag (e.g. due to sanctions on their home country). They appear in the data with pseudo-country codes like `C01`, `C02`, etc.
- `isIndividualContestant(code)` in `utils/data.ts` detects these codes via the regex `/^C\d+$/`.
- These codes are **filtered out** in several places: `CountryLink.astro` hides them from country links, `results_country.aspx.astro` excludes them from the country results table, and `results_matrix.aspx.astro` excludes them from the matrix.
- When adding new pages or views that list countries, use `isIndividualContestant()` to decide whether to show or hide these entries.

### Sub-Navigation
- Groups of related pages (country pages, year pages, results pages) share a pill-nav bar via `PageHeader.astro`.
- Nav item lists are generated by helpers in `utils/nav.ts`: `countryNavItems(code)`, `yearNavItems(year)`, `resultsNavItems`.
- When adding a new page to a group, update the relevant function in `nav.ts` so the pill-nav includes it.

## ⌘K Command Palette — IMPORTANT

The site has a global ⌘K search palette (`components/CommandPalette.astro`). It builds a client-side search index from JSON data and a hardcoded `STATIC_PAGES` array. **You must keep it in sync when making changes:**

### When to update the Command Palette
1. **Adding a new page** — add it to the `STATIC_PAGES` array (for top-level pages) or ensure the new entity type is covered by the dynamic index-building logic.
2. **Renaming/removing a page** — update or remove its entry in `STATIC_PAGES` and fix any `href` patterns in the dynamic loops.
3. **Adding a new data entity type** (e.g. a new kind of searchable item) — add a new loop in `loadSearchIndex()` that pushes `SearchItem` entries, and add a corresponding type to the `SearchItem.type` union + icon mappings + `PREFIXES`.

### How it works
- `STATIC_PAGES` (hardcoded array): top-level navigational pages like Timeline, Countries, Hall of Fame.
- Dynamic entries: built at runtime from `countries.json`, `participants.json`, `timeline.json`, and `country_results_by_year.json`. Each country generates 4 entries (info, individual, team, hall of fame). Each year generates 4 entries (info, country results, individual results, statistics). Each participant gets 1 entry. Each country-at-year combination gets 1 entry.
- Search prefixes: `p:` (people), `c:` (countries), `y:` (years), `pg:` (pages).
- Navigation uses `astro:transitions/client`'s `navigate()` for SPA-like transitions.
- The `setup()` function uses an `AbortController` to prevent listener accumulation across View Transitions. It re-runs on `astro:after-swap`.

## View Transitions (SPA-like Navigation)

The site uses Astro's `ClientRouter` (View Transitions API) for SPA-like client-side navigation. This has implications for inline scripts:

### Re-running scripts after navigation
- `<script>` tags (non-inline, module-style) run once and persist across navigations. Use `document.addEventListener('astro:after-swap', setup)` to re-bind DOM listeners after page swaps.
- `<script is:inline define:vars={...}>` tags re-execute on every navigation by default. Use `document.addEventListener('astro:page-load', drawFn)` to also redraw on back/forward navigation.
- Always clean up listeners with `AbortController` if re-binding on `astro:after-swap` to avoid listener accumulation.

### Common pitfall
If you add interactive behavior via `<script>` and it stops working after navigating to/from the page, you likely need to listen to `astro:page-load` or `astro:after-swap`.

## Chart System

Charts are drawn with the Canvas 2D API using shared helper functions.

### Architecture
- `public/js/chart-utils.js` — global helper functions loaded via `<script is:inline src="/js/chart-utils.js">` in Layout.astro. Provides: `initCanvas`, `chartSetup`, `drawDistChart`, `drawMedalStackedChart`, `drawLineChart`, `drawMultiLine`, `drawRankOverlay`, `drawYGrid`, `drawXLabels`, `setupTooltip`, `awardColor`, `drawScatterPlot`, etc.
- Each page's `<script is:inline define:vars={...}>` block calls these helpers, passing serialized JSON data via `define:vars`.
- Chart colors come from `utils/chartColors.ts` (server-side) and are passed to the client as serialized JSON.

### Adding a new chart
1. Serialize your data in the page frontmatter and pass it via `define:vars`.
2. Call existing helpers from `chart-utils.js` or add new ones there.
3. Wrap the drawing call in a named function, call it immediately, and also register it on `astro:page-load`.
4. Use the `.chart-wrapper` CSS class on the containing `<div>`.
5. Use `ChartLegend.astro` for the legend below the chart.

## CSS Guidelines

### Single Global Stylesheet
All shared styles live in `src/styles/global.css`. Do not create additional CSS files.

### CSS Custom Properties
Use the design tokens defined in `:root` — never hard-code colors, fonts, radii, or shadows. Key tokens:
- **Colors:** `--navy`, `--gold`, `--cream`, `--link`, `--text`, `--text-secondary`
- **Medals:** `--medal-gold`, `--medal-silver`, `--medal-bronze`, `--medal-hm` (and `*-bg`/`*-fg` variants)
- **Chrome:** `--backdrop`, `--kbd-bg`, `--kbd-border`, `--icon-bg`
- **Typography:** `--font-display` (headings), `--font-body` (everything else)
- **Spacing/Layout:** `--radius`, `--max-width`, `--shadow-sm`, `--shadow`, `--shadow-md`, `--transition`

### Scoped vs Global
- **Global CSS** (`global.css`): shared primitives — tables (`.data-table`), awards (`.award-gold-medal`), badges (`.medal-badge`), cards, buttons, forms, page content typography, chart wrappers, pill pickers (`.problem-picker`, `.problem-pill`), scatter layouts (`.scatter-grid`, `.scatter-cell`, `.scatter-corr`), chart notes (`.chart-note`).
- **Scoped `<style>`** (in `.astro` files): component/page-specific layout that won't be reused elsewhere. Header/footer styles are scoped inside `Layout.astro`.

### Naming Conventions
- Flat BEM-like class names: `.data-table`, `.chart-wrapper`, `.medal-badge`, `.pill-nav`.
- Modifier pattern: `.btn--pill`, `.btn--ghost`, `.notice--warning`, `.badge--neutral`.
- Utility classes: `.text-right`, `.text-center`.

### Tables
Use `.data-table` with sticky headers, striped rows, and hover. Sortable columns get `.sortable-header`. Use `utils/sort.ts` helpers for sort URL generation and indicators.

### Responsive
- Breakpoint at 768px.
- Put responsive overrides in a `@media (max-width: 768px)` block at the end of the relevant style section.

## Checklist: Adding a New Page

1. Create `src/pages/<name>.aspx.astro` (use `.aspx.astro` suffix to match imo-official.org URLs).
2. Wrap content in `<Layout title="..." description="..." path="/...">`.
3. If the page belongs to a nav group (country/year/results), add it to the relevant function in `utils/nav.ts`.
4. **Update the Command Palette** — add to `STATIC_PAGES` in `CommandPalette.astro` if it's a top-level page, or ensure the dynamic index covers it.
5. If the page has a header nav, add the `<script>` to the header nav and update any existing nav items in `Layout.astro`'s `NAV_ITEMS` if needed.
6. Add data loader to `utils/data.ts` if new JSON is needed.
7. Run `npm run check` to verify types.
